# Architecture — mnsk7-tools.pl

*(Выход агента 03_wp_architect; актуализировано 2026-03-06)*

Подход к теме, шаблоны Woo, данные, плагины, кеш. Только дизайн системы, без кода. Источники: REQUIREMENTS, SEO_PLAN, AS_IS_AUDIT.

---

## 1. Подход к теме

- **Текущее состояние:** parent Storefront (WooCommerce), child mnsk7-storefront. Child сохраняем — правки не теряются при обновлении parent.
- **Решение:** продолжать использовать **child theme**. Не переходить на кастом-тему с нуля — слишком дорого при уже работающем стейдже и деплое.
- **Правило:** все кастомизации интерфейса и Woo — через overrides и хуки в child или в отдельном плагине проекта; не править файлы parent и плагинов напрямую (wp_theme_architecture).

---

## 2. Где правим шаблоны Woo (overrides)

- **Путь overrides:** в **child-theme** `mnsk7-storefront/woocommerce/`. Копировать только нужные шаблоны из `storefront/` или из `woocommerce/templates/`.
- **Соглашения:**
  - Имена и иерархия файлов — как в Woo (например `content-single-product.php`, `archive-product.php`, `single-product/...`).
  - В шаблоне — только разметка и вызовы Woo-функций; бизнес-логику выносить в плагин или `inc/` (не смешивать логику в шаблонах).
- **Приоритетные overrides (по REQUIREMENTS и спринтам):**
  - **Карточка товара (single product):** блок ключевых параметров, «подходит для», наличие/доставка (Sprint 02). Файлы: `single-product/` при необходимости, либо `content-single-product.php`.
  - **Страница категории (archive):** разметка списка товаров и блока фильтров, при необходимости блок текста/FAQ (SEO_PLAN, CONTENT_PLAN). Файл: `archive-product.php` и связанные.
- **Assets:** CSS/JS кастомные — в `mnsk7-storefront/assets/css`, `assets/js`; подключать через `functions.php` или модуль в `inc/`, не хардкодить в parent.

---

## 3. Кастомная логика (не в шаблонах)

- **Вынести из `functions.php` child-theme** в:
  - **mu-plugin** проекта (например `mnsk7-tools.php` или аналог в `wp-content/mu-plugins/`) — логика, общая для темы и Woo (фильтры, хелперы, регистрация блоков).
  - Либо **обычный плагин** в `wp-content/plugins/` с префиксом проекта — если нужна активация/деактивация.
- **В child `functions.php`** оставить только: подключение parent, подключение модулей из `inc/` (если логика остаётся в теме), настройки темы (цвета, шрифты через `theme customizer`).
- **Правило:** при смене темы бизнес-логика в mu-plugin/плагине не теряется (AS_IS_BACKLOG P1-06).

## 4. Данные: что где храним

### 4.1 Атрибуты товаров (WooCommerce product attributes)

- **Источник истины** для фильтров и отображения в карточке: глобальные атрибуты WooCommerce (Products → Attributes).
- **Минимум по content_catalog_rules и DISCOVERY:** материал обработки, тип операции, диаметр, длина (рабочая/общая), хвостовик/цанга, покрытие, количество зубьев. В AS_IS уже есть: srednica, fi (trzpienia), dlugosc-robocza, dlugosc-calkowita, zastosowanie, typ и др. — привести к единым именам и заполняемости (аудит P1-02).
- **Использование:** фильтр (один выбранный плагин), блок «ключевые параметры» и «подходит для» в карточке — вывод атрибутов из продукта. Никаких дублей в произвольных meta без необходимости.

### 4.2 Таксономии

- **product_cat** (категории товаров): структура по типу инструмента и при необходимости по материалу (SEO_PLAN §2). Не дублировать «тип» и «материал» в одной плоской куче — либо иерархия, либо атрибуты + фильтры + лендинги.
- **product_tag** — по необходимости для тегов (хиты, «bestseller» и т.д.), без перегруза.
- Кастомные таксономии не вводить без обоснования; атрибуты + категории достаточно для фильтрации и SEO.

### 4.3 Метаполя товаров

- Стандартные Woo: `_sku`, `_stock`, цена и т.д. — без дублирования.
- Дополнительные поля только если нет атрибута (например дата поступления, рейтинг из внешней системы). Предпочтение — атрибутам для всего, что участвует в фильтрах и отображении.

### 4.4 Медиа (изображения)

- Хранение в `uploads/`; alt и title — в медиабиблиотеке WP (post meta вложений). Массовое заполнение alt по правилу: ключевое слово + параметры фрезы (SEO_PLAN, P2-04). Имена файлов — хэши; не полагаться на имя файла для SEO.

## 5. Стратегия плагинов

- **Минимум плагинов.** Оставлять только необходимые для оплаты, доставки, фильтрации, SEO, безопасности, кеша, аналитики (AS_IS_AUDIT).
- **Критичные (оставить):** WooCommerce, Yoast SEO, Przelewy24 + Przelewy24 Raty при необходимости (разные шлюзы), один кеш (LiteSpeed или WP Rocket), InPost/Paczkomaty, limit-login-attempts-reloaded, Google Site Kit, один GTM, один Facebook Pixel.
- **Один плагин на одну функцию:** один фильтр (остальные удалить после миграции URL/редиректов), один вишлист, один page builder (если нужен), один плагин профилей. Schema — только Yoast (убрать schema-and-structured-data-for-wp).
- **Кастом:** mu-plugin проекта (staging-safety уже есть; добавить mnsk7-tools или аналог для бизнес-логики). Не плодить мелкие плагины без причины.

---

## 6. Кеш и производительность

- **Страничный кеш:** один движок — либо LiteSpeed Cache, либо WP Rocket. Остальные (Seraphinite, второй из пары) отключить. Исключения: `cart`, `checkout`, `my-account` не кешировать (cookie, динамический контент).
- **Object cache:** Redis уже подключён (object-cache.php) — сохранить для запросов к БД.
- **Изображения:** конвертация в WebP и сжатие (массово или через плагин/скрипт) — отдельная задача; без этого LCP остаётся риском (AS_IS_AUDIT D).
- **JS/CSS:** минимизация и объединение — средствами выбранного кеш-плагина; критичный CSS выше сгиба для LCP.

---

## 7. Безопасность (базовый уровень)

- **xmlrpc.php:** закрыть (403 или отключить) через .htaccess или плагин (AS_IS_RISKS, P0-03).
- **Роли:** не расширять без нужды; при необходимости — user-role-editor уже есть.
- **Бэкапы:** наличие расписания (БД + файлы); при отсутствии — внедрить (UpdraftPlus или аналог).
- **Обновления:** ядро WP и плагины по политике проекта; тестирование на стейдже перед продом.
